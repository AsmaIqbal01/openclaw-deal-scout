# Implementation Plan: Pipeline Orchestration, Error Handling & End-to-End Wiring

**Branch**: `004-pipeline-orchestration` | **Date**: 2026-07-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-pipeline-orchestration/spec.md`

## Summary

Wire Gmail-intake (step 1), HubSpot CRM-logger (step 2), and Discord-notifier (step 3) into a single scheduled pipeline cycle implemented as a new `pipeline_orchestrator` Python package. The orchestrator adds a concurrent-cycle lock, a SIGTERM-safe shutdown path, per-entry `crm_status` / `notify_status` retry tracking, startup config validation, and a rotating cycle-summary log — all within the zero-cost, zero-infrastructure constraints of the project constitution.

---

## Technical Context

**Language/Version**: Python 3.12 (matches existing package venv)
**Primary Dependencies**:
- `gmail_intake.server.check_new_deals_handler` — async coroutine; called via `asyncio.run()`
- `crm_logger.server.sync_deals_to_crm` — synchronous
- `discord_notifier.server.sync_notifications` — synchronous
- `google-auth` — exception type `google.auth.exceptions.RefreshError` caught in runner
- `gmail_intake.models.RateLimitExhaustedError` — Gemini quota signal
- stdlib only: `asyncio`, `signal`, `tempfile`, `json`, `logging`, `time`, `pathlib`, `os`, `datetime`, `dataclasses`

**Storage**: File-based JSON (`processed_ids.json`, path from `STATE_STORE_PATH`) — existing shared state store; `.pipeline.lock` in same directory; `pipeline.log` rotating file in same directory (or `PIPELINE_LOG_PATH`)

**Testing**: `pytest`, `pytest-asyncio` (asyncio_mode=auto)

**Target Platform**: Ubuntu 22.04 (WSL2 + native Linux); systemd 249+ for timer-based scheduling

**Project Type**: Single package (`src/pipeline_orchestrator/`) alongside existing `src/` packages

**Performance Goals**:
- Cycle starts within ±1 minute of configured `POLL_INTERVAL_MINUTES`
- Lock check (staleness evaluation) completes within one file read
- Cycle summary emitted within the `finally` block — always fires, even on exceptions

**Constraints**:
- No new paid dependencies (Constitution Principle I)
- Steps 1–3 internals unchanged — orchestrator is a wiring layer only
- All credentials via static env vars (Constitution Principle III)
- Lock file must use UTC ISO-8601 timestamps to support timezone-change staleness checks

## Architecture Decisions

All decisions recorded in full at [research.md](research.md). Summary:

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Dual-mode scheduler (`SCHEDULER_MODE=loop\|systemd`) | systemd for prod; sleep-loop for dev/test. Default: `loop` (safe for first run). |
| 2 | `src/pipeline_orchestrator/` new package | Steps 1-3 untouched; orchestrator is an independently testable wiring layer. |
| 3 | `python -m pipeline_orchestrator` entry point | systemd mode: one-shot invocation; loop mode: `while True: run_cycle(); sleep()`. Both share `run_cycle()`. |
| 4 | `PIPELINE_LOG_PATH` with default `<state_dir>/pipeline.log` | Colocates all operational files; stdlib `RotatingFileHandler` = zero cost. |
| 5 | FR-022: catch `RateLimitExhaustedError`, continue steps 2+3 | Step 1 already wrote classified entries; step 2 drains them in the same cycle at no extra cost. |
| 6 | WSL2 systemd: `/etc/wsl.conf [boot] systemd=true` | Operator's WSL2 Ubuntu 22.04 supports systemd natively. |
| 7 | Retry counter stored in state store (`crm_retry_count`, `notify_retry_count`) | Only durable option; in-memory counters lost on restart; avoids a separate lock/file. |

## Interface Contracts

### Step Invocation Interface

| Step | Import path | Function | Calling convention |
|------|------------|----------|--------------------|
| 1 — Gmail Intake | `gmail_intake.server` | `check_new_deals_handler()` | `asyncio.run()` — async |
| 2 — CRM Logger | `crm_logger.server` | `sync_deals_to_crm()` | Synchronous call |
| 3 — Discord Notifier | `discord_notifier.server` | `sync_notifications()` | Synchronous call |

### Cycle Lock File

- **Path**: `<STATE_STORE_DIR>/.pipeline.lock`
- **Content**: Single ISO-8601 UTC line, e.g. `2026-07-22T14:30:00Z`
- **Created**: before any step runs (inside `CycleLock.__enter__`)
- **Deleted**: always in `CycleLock.__exit__` (finally-equivalent)
- **Staleness**: `now - lock_timestamp > LOCK_TIMEOUT_MINUTES`

### Cycle Summary JSON

```json
{"ts": "2026-07-22T14:30:05Z", "emails_processed": 3, "crm_logged": 1, "notified": 1, "pending": 0, "errors": []}
```

All canonical `errors` tokens defined in spec Interface Contracts section.

### Environment Variables

| Variable | Default | Validation |
|---|---|---|
| `STATE_STORE_PATH` | required | Non-empty string → `Path` |
| `SCHEDULER_MODE` | `loop` | `"loop"` or `"systemd"` |
| `POLL_INTERVAL_MINUTES` | `15` | Integer ≥ 1 |
| `LOCK_TIMEOUT_MINUTES` | `30` | Integer ≥ 1 |
| `PIPELINE_LOG_PATH` | `<state_dir>/pipeline.log` | Non-empty string → `Path` |
| `LOG_MAX_BYTES` | `10485760` | Integer ≥ 1 |
| `LOG_BACKUP_COUNT` | `3` | Integer ≥ 0 |
| `MAX_PENDING_RETRIES` | `10` | Integer ≥ 1 |

### State Store Extension

Two new per-entry fields added by the orchestrator (written to `processed_ids.json` after each step):

```json
{
  "gmail_message_id": "...",
  "outcome": "deal_extracted",
  "crm_status": "logged | pending | failed",
  "crm_retry_count": 0,
  "notify_status": "sent | pending | failed",
  "notify_retry_count": 0
}
```

`crm_status` / `notify_status` absent on non-deal entries (`not_a_deal`, `body_absent`, etc.).
